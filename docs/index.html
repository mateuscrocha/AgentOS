AgentOS

Sistema Operacional de Agentes de IA

Assim como um sistema operacional gerencia processos, memória e comunicação entre programas, o AgentOS gerencia agentes, memória e comunicação entre eles. Construído sobre o Claude Code e Gemini CLI.

v0.9.0 19 Agentes 42 Skills 5 Protocolos
⚙
Padronização
Todos os agentes seguem o mesmo formato e convenções, tornando o sistema previsível e fácil de manter.
🔒
Isolamento
Cada space é um namespace isolado. Agentes de spaces diferentes não interferem entre si.
🧠
Memória Persistente
Agentes têm memória estruturada que persiste entre sessões, com escopos hierárquicos.
👥
Colaboração
Comunicação síncrona (invocação direta) e assíncrona (handoff e bus de eventos) entre agentes.
🚀
Extensibilidade
Novos agentes, skills e times podem ser adicionados a qualquer momento sem afetar o existente.
🌐
Multi-Runtime
Suporte completo ao Claude Code e Gemini CLI com dual-write automático.

Arquitetura

Duas Camadas

O AgentOS é organizado em duas camadas distintas: o sistema (core do OS) e os spaces (espaços do usuário).

⚙
Camada do Sistema
system/ — Core do AgentOS. Contém agentes de sistema, protocolos, templates, skills globais e memória global. Não deve ser modificada diretamente.
📁
Camada do Usuário
spaces/ — Espaços criados pelo usuário. Cada space é um namespace isolado, subdividido em áreas e times.

Passo a Passo

1

Bootstrap do sistema

Inicializa o AgentOS com 19 agentes, 5 protocolos e 42 skills.

bash 
/setup
2

Verificar o sistema

Confirme que o sistema está ativo e os agentes funcionando.

bash 
/status
3

Criar um space

Crie seu primeiro namespace isolado.

bash 
/new-space meu-projeto
spaces/meu-projeto/ ├── SPACE.md ← Manifesto do space ├── memory/ │ ├── world.md ← Estado do space │ └── handoff.md ← Tarefas pendentes ├── guidelines/ │ └── GUIDELINES.md ← Documentação do space └── areas/ ← Vazio (ainda sem áreas)
4

Criar uma área

Subdivida o space em domínios lógicos.

bash 
/new-area meu-projeto backend
5

Criar um agente

Crie sua unidade fundamental de trabalho. O sistema perguntará sobre persona, capacidades e regras.

bash 
/new-agent meu-projeto backend assistente
spaces/meu-projeto/areas/backend/agents/assistente/ ├── AGENT.md ← Definição do agente ├── memory/ │ └── history.md ← Histórico de ações └── skills/ ← Vazio (ainda sem skills) .claude/agents/meu-projeto--backend--assistente.md ← Integração Claude Code .gemini/agents/meu-projeto--backend--assistente.md ← Integração Gemini CLI
6

Adicionar uma skill

Dê ao agente um processo documentado e repetível.

bash 
/new-skill meu-projeto/backend/assistente revisar-codigo
7

Interagir com o agente

Pronto! O agente lê sua memória, executa a tarefa e atualiza o contexto.

Claude Code 
"Use o assistente para revisar o último PR"
Gemini CLI 
@meu-projeto--backend--assistente revise este código

Conceitos Essenciais

Os blocos fundamentais que formam o AgentOS.

🏢
Space
Namespace isolado de nível superior. Equivale a um projeto ou domínio de trabalho. Agentes de spaces diferentes não se comunicam. Exemplos: automacao-vendas, analise-dados.
📁
Area
Subdivisão temática dentro de um space. Agrupa agentes e times relacionados. Exemplos: backend, frontend, dados.
🤖
Agent
Unidade fundamental. Tem Persona (quem é), Capacidades (o que faz), Skills (processos) e Memória (contexto). Regra de ouro: um agente, uma responsabilidade.
👥
Team
Grupo de agentes colaborativos dentro de uma área. Tem memória compartilhada e um agente líder que coordena os demais.
⚡
Skill
Processo documentado passo-a-passo que um agente executa. Sem skill, o agente improvisa. Com skill, o comportamento é previsível e repetível.
🧠
Memory
Memória hierárquica em 5 níveis. Arquivos: world.md (estado atual), history.md (log de ações), handoff.md (tarefas transferidas).

Sistema de Memória

Toda memória é baseada em arquivos markdown com estrutura definida e regras rígidas de escopo.

Hierarquia de Escopos

system/memory/world.md ← Estado global (todos leem) └── spaces/{space}/memory/world.md ← Estado do space └── areas/{area}/memory/world.md ← Estado da área └── teams/{time}/memory/world.md ← Estado do time └── {agente}/memory/history.md ← Histórico do agente

Tabela de Escopos

EscopoLocalizaçãoQuem LêQuem Escreve
Sistemasystem/memory/Todos os agentesAgentes do sistema
Spacespaces/{space}/memory/Agentes do space + sistemaAgentes do space + sistema
Área.../areas/{area}/memory/Agentes da área + sistemaAgentes da área + sistema
Time.../teams/{time}/memory/Membros do time + sistemaMembros do time + sistema
Agente{agente}/memory/Próprio + memory-managerApenas o próprio agente

Tipos de Arquivo

world.md
Snapshot do estado atual. Pode ser reescrito completamente. Obrigatório incluir seção "Última Alteração".
reescrevível
history.md
Log cronológico de ações do agente. Somente adiciona ao final, nunca substitui.
append-only
handoff.md
Tarefas transferidas entre agentes. Entradas adicionadas; concluídas removidas no cleanup.
append + cleanup
bus.md
Log de eventos broadcast do sistema. Qualquer agente lê; sistema escreve. Rotação a 50 entradas.
broadcast
registry.md
Registros mestres de agentes, skills, times e docs. Mantidos pelos agentes de sistema responsáveis.
sistema apenas

Protocolo de Acesso

Antes de Agir

1. Ler {agente}/memory/history.md        ← contexto pessoal
2. Ler {escopo}/memory/world.md          ← contexto do escopo
3. Ler system/memory/world.md            ← contexto global

Depois de Agir

1. Atualizar {agente}/memory/history.md
2. Se estado mudou → atualizar {escopo}/memory/world.md
3. Se precisa passar trabalho → escrever em {escopo}/memory/handoff.md
4. Se evento relevante → registrar em system/memory/bus.md

markdown

**Data:** 2026-03-23 14:30
**Prioridade:** Normal
**Status:** Pendente

### Contexto
O que estava sendo feito e por que o handoff é necessário.

### Tarefa
O que o agente destino deve fazer.

### Inputs
- Arquivo: `caminho/para/arquivo`
- Dado: informação relevante

### Resultado (preenchido pelo destino)
Output produzido pelo agente destino.

Vamos criar um cenário completo e realista — automatizar coleta, análise e geração de relatórios.

bash

/new-area analise-dados pipeline

2. Criar os Agentes

📥
coletor
Persona: Especialista em coleta e integração de dados.
Skills: Ler CSVs, conectar APIs, validar dados, limpar duplicatas.
📈
analista
Persona: Cientista de dados senior.
Skills: Análise estatística, identificar padrões, gerar insights.
📃
relator
Persona: Especialista em comunicação de dados.

Guia de Desenvolvimento

Convenções internas e como estender o AgentOS corretamente.

Princípios Fundamentais

Tudo é Arquivo
Memória, agentes, skills, protocolos — nada existe fora do sistema de arquivos.
Escopo Explícito
Toda operação tem um escopo definido. Nunca escreva fora do seu escopo.
Templates First
Sempre use os templates em system/templates/ como base para criar recursos.
Registre Tudo
Toda criação/mudança relevante vai para o registry e o bus de eventos.
Leia Antes de Agir
Agentes sempre leem sua memória antes de executar qualquer tarefa.
Nunca Invente
Agentes documentam e executam apenas o que está definido no sistema.

Convenções de Nomes

Todos os nomes devem ser kebab-case: letras minúsculas, sem espaços, sem underscores. Apenas letras, números e hífens.

Agente trabalha sozinho em tarefas independentesAgente direto na área
2+ agentes precisam trocar informaçõesCrie um time
Fluxo sequencial (A → B → C)Time com handoffs
Agentes compartilham contexto/estadoTime com memória compartilhada

Peça ao sistema para evoluir o agente:

"Evolua o agente assistente do backend do meu-projeto para incluir capacidade de testes"

O agent-manager atualiza a definição mantendo a memória e skills existentes, e incrementa a versão.

O health-monitor cria automaticamente handoffs para os agentes responsáveis corrigirem os problemas encontrados. Na maioria dos casos, os problemas são resolvidos automaticamente na próxima interação.
